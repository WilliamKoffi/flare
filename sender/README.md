# Prospect Mailer

Prospect Mailer is a small Rust command-line application for sending personalized
outreach emails through the Gmail API. It reads prospects from TOML and mail
settings from a runtime `.env` file, renders a Markdown message as plain text, adds
a personalized website URL, throttles
deliveries, and records successful sends to avoid contacting the same prospect
twice.

The repository also contains an optional Playwright scraper for collecting raw
lawyer directory records. Its output must currently be transformed into the
mailer's prospect format before it can be used.

## Features

- Runtime-configurable email subject and Markdown body
- Authenticated encryption for all personalized link data
- Optional lnk.ua shortening of encrypted links
- Gmail API delivery with OAuth 2.0
- Dry-run mode for reviewing messages without sending them
- Random delay between successful sends
- Per-run sending cap
- Persistent ledger that prevents duplicate sends

## Requirements

- Rust toolchain with Cargo
- A Google Cloud project with the Gmail API enabled
- OAuth desktop application credentials
- A Gmail account authorized to send messages
- Optional scraper: Node.js or Bun, plus Playwright and Chromium

```text

├── src/
│   ├── main.rs       # CLI and sending workflow
│   ├── prospect.rs   # Prospect loading and personalized URL generation

│   ├── client.rs     # Gmail OAuth authentication and message delivery

│   ├── ledger.rs     # Successful-send persistence
│   └── template.rs   # Subject and body interpolation
├── scrapper/         # Optional lawyer-directory scraper
└── storage/
    ├── prospects.toml    # Mailer input
    ├── credentials.json  # Gmail OAuth client credentials
    ├── token.json        # Generated OAuth token
    ├── sent.json         # Generated delivery ledger
    └── body.md           # Private message body, ignored by Git
```

## Gmail Setup

1. Create or select a project in Google Cloud Console.
2. Enable the Gmail API.
3. Configure the OAuth consent screen.
4. Create OAuth credentials for a desktop application.
5. Download the credentials and save them as `storage/credentials.json`.

On the first real send, the application opens the OAuth authorization flow and
stores the resulting token in `storage/token.json`. Runtime files in `storage/`
are ignored by Git.

The value passed to `--from` should match the Gmail account authorized during
this flow.

## Prospect Format

The mailer expects TOML entries:

```toml
[[lawyer]]
id = "kouassi-aya-2026-05"
name = "Aya Kouassi"
email = "aya@example.com"
gender = "F"
phone = "0700000000"
image = "https://example.com/aya.jpg"
firm = "Cabinet Kouassi"
url = "https://example.com/presentation/v1_..."
```

Fields:

- `id`: stable unique identifier used by the send ledger
- `name`: required recipient name
- `email`: required valid recipient email address
- `gender`: required; must be `M` or `F`
- `phone`, `image`, `firm`: optional personalized URL values
- `url`: generated by `prepare-links` and used by `send` and `simulate`

Other directory fields such as `link`, `office`, and `valid` are preserved but
ignored by the mailer.

The `prepare-links` command serializes the prospect fields as JSON, encrypts
them, and writes the resulting URL back to each `[[prospect]]` or `[[lawyer]]`
record. Existing `url` values are replaced. Required values are always included
in the encrypted payload; optional values are omitted when blank:

```text
<base-url>/v1_<base64url-nonce-ciphertext-and-tag>
```

The mailer uses AES-256-GCM with a random 12-byte nonce. Its 256-bit key is the
SHA-256 digest of `LINK_ENCRYPTION_TOKEN`. The encoded bytes are the nonce
followed by the ciphertext and 16-byte authentication tag. The web application
must use the same token and format to decrypt the final path segment. This keeps
the prospect data out of lnk.ua, browser history, analytics URLs, and referrer
headers.

## Mail Configuration

Generate the private runtime files:

```bash
cargo run -- setup
```

The command creates `.env`, `storage/body.md`, `storage/prospects.toml`, and
`storage/sent.json` when they do not exist. Existing files are never
overwritten. Google OAuth credentials must still be downloaded to
`storage/credentials.json`; `storage/token.json` is generated during the first
OAuth authorization.

Edit `.env` to configure the website URL, sender address, subject, and body:

```dotenv
BASE_URL="https://example.com/presentation"
SENDER_EMAIL="sender@example.com"
LINK_ENCRYPTION_TOKEN="replace-with-at-least-32-random-characters"
LNK_BEARER_TOKEN=""
MAIL_SUBJECT="Une idée de vitrine web pour votre cabinet — Maître {{name}}"
MAIL_BODY_PATH="storage/body.md"
TEST_EMAIL="score-address@example.com"
TEST_EMAIL_SLOTS=2
```

`LINK_ENCRYPTION_TOKEN` is required and must be at least 32 bytes. Generate a
high-entropy value and configure the exact same value in the web application.
Keep it secret and stable: changing it makes previously generated links
undecryptable.

Set `LNK_BEARER_TOKEN` to the token from your lnk.ua account when using
`prepare-links --shorten`. The command sends only the encrypted URL to
`POST https://lnk.ua/api/v1/link/create`, then stores the returned short URL in
the prospect record. Without `--shorten`, it stores the encrypted long URL. See the
[lnk.ua Link Shortening API documentation](https://lnk.ua/docs/#/en/link).

Edit `storage/body.md` to change the body. Both private files are ignored by
Git and are loaded at runtime, so they can be changed after compiling the
application. The subject and body support these placeholders:

```text
{{name}}
{{link}}
```

Messages are sent as UTF-8 plain text, not rendered HTML.

## Usage

Build the application:

```bash
cargo build
```

Export the unique email addresses from the prospect TOML to a CSV suitable for
bulk verification:

```bash
cargo run -- export-emails
```

The default output is `storage/prospects-emails.csv`. Override either path when
needed:

```bash
cargo run -- export-emails \
  --prospects storage/prospects.toml \
  --output storage/prospects-emails.csv
```

Generate or replace the encrypted `url` field for every prospect:

```bash
cargo run -- prepare-links
```

Generate the encrypted URLs and shorten them through lnk.ua before storing them:

```bash
cargo run -- prepare-links --shorten
```

The file is updated only after every URL has been generated successfully. Run
this command again whenever personalized prospect data changes.

Review the prepared messages without sending:

```bash
cargo run -- simulate
```

Send the prepared campaign:

```bash
cargo run --release -- send
```

The `send` and `simulate` commands require every prospect to have a non-empty
`url`. They never generate, encrypt, or shorten URLs.

Set `TEST_EMAIL` and `TEST_EMAIL_SLOTS` in `.env` to send plain-text test
messages before prospect messages. Test messages use an unrelated subject and
body, consume daily-cap slots, and are recorded in the ledger. For example,
with a daily cap of 15 and 2 test slots, at most 13 prospect messages are sent.
Set `TEST_EMAIL_SLOTS=0` to disable test messages.

Inspect command-specific options with:

```bash
cargo run -- prepare-links --help
cargo run -- send --help
```

The minimum delay must not be greater than the maximum delay.

## Delivery Ledger

After Gmail accepts a message, the prospect ID and UTC timestamp are written to
`storage/sent.json`. Future runs skip IDs already present in that file.

The `--capacity` option includes successful sends already recorded for the
current UTC calendar day. Dry runs apply the same cap without modifying the
ledger.

Use a separate ledger path for testing if you do not want to affect the normal
send history:

```bash
cargo run -- simulate \
  --ledger "storage/test-sent.json"
```

## Optional Scraper

`scraper/main.ts` uses Playwright to browse the configured lawyer directory,
follow pagination, and write records to `scrapper/directory.json` by default:

```bash
cd scraper
npm install
npm run browsers
npm run scrape
```

Pass a different output path as the first argument:

```bash
npm run scrape -- ../scrapper/another-file.json
```

The scraper produces TOML records that the mailer can read directly, but the
directory does not currently expose gender. Add `gender = "M"` or
`gender = "F"` to each record before sending. Records missing required fields
are rejected before authentication or delivery.

The scraper currently launches Chromium in visible mode and targets selectors
specific to the configured directory. Website changes may require selector
updates.

## Safety Notes

- Keep `storage/credentials.json`, `storage/token.json`, prospects, and send
  ledgers private.
- Use `--dry-run` before every campaign.
- Use stable prospect IDs; changing an ID can bypass duplicate protection.
- Confirm recipient consent and comply with applicable privacy, anti-spam, and
  professional solicitation rules.
- Start with conservative caps and delays.
